\begin{abstract}
Kubernetes upgrades are recurring software evolution activities in which platform changes interact with application APIs, disruption budgets, admission controls, workload availability assumptions, scheduling capacity, operators, add-ons, and observability capabilities. Existing operational tools are useful for detecting deprecated Kubernetes APIs, but upgrade readiness also depends on whether workloads can tolerate node disruption, whether admission policies will allow recreated objects, whether platform engineers can collect sufficient evidence, and whether remediation actions are traceable to concrete cluster state. This paper presents \tool, a Kubernetes-native bounded-write operator artifact for evidence-guided upgrade readiness assessment. The artifact implements two custom resources, \crd{UpgradeAssessment} and \crd{UpgradePlan}, and deterministic read-only checkers that produce structured findings, evidence records, aggregate risk scores, non-executing decisions, and prioritized remediation actions. The operator does not upgrade clusters, drain nodes, patch workloads, or apply remediations; it reads assessed resources and writes only its own assessment status and generated plan resources. We report implementation fidelity, local validation from \code{go test ./...}, and a controlled Kind benchmark with 18 labeled findings across deprecated API, workload availability, readiness, PDB, admission, and policy-risk families. In this controlled benchmark, \tool{} matched all labeled findings with no false positives or false negatives, while Pluto and kubent matched the three deprecated-API findings but did not cover the 15 non-API upgrade-readiness findings. The broader production-oriented empirical study remains specified as a reproducible protocol covering managed AKS validation, expert review, optional field studies, resource metrics, and scoring-sensitivity analysis. Future explanation support is discussed but remains outside the evaluated artifact.
\end{abstract}

\section{Introduction}
Kubernetes has become a common substrate for operating cloud-native software systems, but a production cluster is not merely an execution platform. It is a continuously evolving socio-technical system composed of workloads, controllers, custom resources, admission policies, resource constraints, identity rules, observability pipelines, and managed-provider services. Upgrading such a system is a software evolution problem: a platform version change can alter the validity, behavior, or operational assumptions of dependent artifacts \cite{lehman1980,mens2008,bohner1996}. The practical challenge is not only whether the target Kubernetes version exists. The challenge is whether the specific cluster can be upgraded without exposing application availability, policy, capacity, compatibility, or observability gaps.

The Kubernetes ecosystem already provides authoritative upgrade primitives and policies. The Kubernetes documentation defines API deprecation, removed APIs, version-skew constraints, API-initiated eviction, PodDisruptionBudget behavior, admission webhooks, ValidatingAdmissionPolicy, custom resources, RBAC, and the operator pattern \cite{kubernetesDeprecationPolicy,kubernetesDeprecationGuide,kubernetesVersionSkew,kubernetesEviction,kubernetesDisruptions,kubernetesAdmissionWebhooks,kubernetesValidatingAdmissionPolicy,kubernetesCustomResources,kubernetesRbac,kubernetesOperatorPattern}. Managed platforms such as AKS add provider-specific upgrade options, node-image mechanics, upgrade channels, and day-2 practices \cite{aksUpgradeOptions,aksNodeImage,aksDay2}. These documents are necessary, but they do not by themselves turn a live cluster into a structured, evidence-backed readiness assessment.

Operational teams often complement official guidance with tools such as Pluto, kube-no-trouble (kubent), Helm API migration documentation, and Helm metadata repair tools \cite{pluto,kubent,helmKubernetesApis,helmMapKubeApis}. These tools are valuable, especially for API removals, but they focus on a subset of readiness risk. A cluster can have no removed API versions and still fail operationally because a single-replica workload is disrupted, a PodDisruptionBudget blocks node drain, an admission webhook is unavailable, a restricted namespace rejects recreated pods, a node pool lacks one-node-loss headroom, or RBAC prevents the platform team from collecting the evidence needed to make a decision.

This paper argues that Kubernetes upgrade readiness should be treated as evidence-guided impact analysis over live cluster state. Rather than producing only warnings or checklist items, an assessment should generate findings that identify affected resources, observable evidence, severity, operational impact, and a non-executing remediation recommendation. Those findings should be materialized in a Kubernetes-native artifact so that platform engineers can review, audit, store, and compare assessments before upgrade execution.

We instantiate this idea in \tool, a read-only-by-design operator artifact. The current implementation exposes \crd{UpgradeAssessment} and \crd{UpgradePlan}; findings and evidence are structured fields inside those resources, not standalone CRDs. The artifact implements deterministic checkers for workload availability, missing readiness probes, PodDisruptionBudget risks, admission webhooks, policy risks, deprecated or removed APIs, capacity drain risk, observability gaps, and RBAC assessment gaps. It computes aggregate scores and decisions, then creates a prioritized \crd{UpgradePlan} without executing any disruptive action.

The paper makes four scoped contributions.
\begin{enumerate}[leftmargin=*]
  \item An evidence-guided model for Kubernetes upgrade readiness assessment that links risk categories to observable evidence and non-executing remediation actions.
  \item A bounded-write Kubernetes operator artifact implementing \crd{UpgradeAssessment}, \crd{UpgradePlan}, deterministic checkers, scoring, decisions, and idempotent plan generation.
  \item An implementation-fidelity analysis that maps manuscript claims to implemented, locally tested, and benchmark-observed artifact behavior.
  \item A reproducible controlled Kind benchmark comparing \tool{} with Pluto and kubent on labeled upgrade-readiness scenarios, plus a protocol for broader managed-cluster and expert-review evaluation.
\end{enumerate}

\section{Background and Problem Statement}
\subsection{Upgrade Readiness as Software Evolution}
Software evolution research frames long-lived software as a system that must change while preserving useful behavior \cite{lehman1980,mens2008}. Change impact analysis studies how to identify the artifacts affected by a proposed change before that change is made \cite{bohner1996}. Kubernetes upgrades fit this pattern: the changed artifact is the platform version, but the affected artifacts include API objects, workloads, admission controllers, add-ons, RBAC permissions, resource requests, and operational workflows.

DevOps and continuous delivery literature emphasizes fast, reliable, automated change delivery, but also documents adoption barriers and the need for reliable feedback loops \cite{humbleFarley2010,chen2015,shahin2017,smeds2015,leite2019,forsgren2018}. SRE practice similarly emphasizes measurable reliability, production readiness, and operational learning \cite{beyer2016,nygard2018}. In a Kubernetes upgrade, readiness assessment is one such feedback loop: it should convert cluster state into evidence that engineers can act on before scheduling disruptive work.

\subsection{Kubernetes-Specific Upgrade Mechanics}
Kubernetes upgrades are constrained by API deprecation and removal schedules, component version skew, eviction behavior, PodDisruptionBudgets, admission control, RBAC, and resource scheduling. The Kubernetes deprecation guide documents removed APIs by release, while the deprecation policy explains the compatibility process \cite{kubernetesDeprecationPolicy,kubernetesDeprecationGuide}. Version-skew policy constrains how far cluster components may differ during upgrades \cite{kubernetesVersionSkew}. API-initiated eviction, including the behavior used by \code{kubectl drain}, interacts with PodDisruptionBudgets and termination behavior \cite{kubernetesEviction,kubernetesDisruptions}. Admission webhooks and ValidatingAdmissionPolicy can reject newly created or updated objects during post-upgrade reconciliation \cite{kubernetesAdmissionWebhooks,kubernetesValidatingAdmissionPolicy}. Pod Security Admission and Pod Security Standards add another policy layer that may reject workload templates after platform or policy changes \cite{kubernetesPodSecurityAdmission}.

\subsection{Problem Statement}
The problem addressed by this paper is:

\emph{Given a Kubernetes cluster and a target Kubernetes minor version, how can platform engineers obtain a repeatable, auditable, evidence-backed readiness assessment that covers more than API deprecations while avoiding unsafe automated remediation?}

A Q1-grade version of this problem requires multiple levels of evidence. First, the system artifact must faithfully implement the claimed model. Second, controlled evaluation must show detection coverage, false positives, and false negatives against labeled scenarios. Third, production-oriented evaluation must assess actionability, transferability, cost, and operational utility in representative clusters. This paper reports the first two levels for a bounded artifact and defines the third level as an evaluation protocol rather than pretending that production generalization has already been established.

The taxonomy is intentionally broader than deprecated APIs. API-removal tools are necessary baselines, but an upgrade may be blocked by disruption semantics, capacity, admission behavior, or insufficient permissions. Treating RBAC denial as an assessment finding is particularly important: an assessment that cannot inspect critical resource types should not silently produce a green result.

The empirical protocol should prioritize depth over breadth. The primary evaluation scope is limited to four risk families: deprecated APIs, workload availability/readiness, PDB and eviction behavior, and admission/policy risks. Capacity, observability, add-on compatibility, and managed-provider constraints remain useful assessment signals, but they should be analyzed as secondary or future extensions until they receive deeper modeling and validation.

\section{Evidence-Guided Assessment Model}
\subsection{Core Data Model}
The model is organized around four concepts.
\begin{description}[leftmargin=*,style=nextline]
  \item[Evidence.] An observable fact collected from the Kubernetes API, such as a replica count, PDB selector, admission webhook failure policy, removed API version, resource request total, or RBAC denial.
  \item[Finding.] A risk or assessment gap backed by one or more evidence records. A finding includes an identifier, type, severity, category, optional Kubernetes resource reference, message, evidence, and recommendation.
  \item[Assessment.] A bounded snapshot of readiness for a target Kubernetes version and namespace scope. In the artifact, this is materialized as \crd{UpgradeAssessment}.
  \item[Plan.] A non-executing remediation artifact that orders required actions and carries the aggregate risk decision. In the artifact, this is materialized as \crd{UpgradePlan}.
\end{description}

\subsection{Scoring and Decisions}
The current artifact uses an intentionally simple scoring model so that reviewers can audit decisions. Critical findings add 25 points, high findings add 10, medium findings add 4, low findings add 1, and informational findings add 0. Aggregate risk is low below 11, medium from 11 to 30, high from 31 to 60, and critical above 60. A single critical finding yields a \code{DoNotUpgrade} decision. A score above 30 yields \code{ProceedWithCaution}. A high or critical RBAC assessment gap can yield \code{AssessmentIncomplete} when higher-priority decision rules do not already apply. Otherwise, the decision is \code{Proceed}.

This score is not presented as a validated risk model. It is a deterministic prioritization mechanism that makes the current artifact inspectable. The thresholds are heuristic and are not empirically calibrated. The empirical evaluation protocol in Section~\ref{sec:evaluation-protocol} is required before claiming calibrated severity, operational utility, or outcome prediction.

\subsection{Scoring Sensitivity Protocol}
Because the current weights are heuristic, the empirical study must include a scoring-sensitivity analysis. The analysis should replay identical findings through at least four decision variants: current weighted scoring, equal weights by finding count, expert-calibrated weights, and a blocker-first model that ignores aggregate score when hard blockers exist. The study should report decision stability, rank-correlation between action orderings, and cases where an upgrade decision changes across variants. Until that analysis is executed, the score should be interpreted as an inspectable implementation mechanism rather than a validated risk estimator.

\subsection{Non-Execution Principle}
\tool{} is designed to support readiness assessment, not upgrade execution. The model forbids the assessment tool from upgrading control planes, draining nodes, patching workloads, modifying policies, changing PDBs, or applying remediations automatically. This follows a conservative SRE posture: the assessment should reduce uncertainty before a risky operation, not become another source of unreviewed change \cite{beyer2016,nygard2018}.

\section{\tool{} Operator}
\subsection{Custom Resources}
The current artifact implements two CRDs.
\begin{itemize}[leftmargin=*]
  \item \crd{UpgradeAssessment}: desired assessment target version, mode, namespace scope, selected checks, and observed status including phase, risk level, score, summary, findings, generated plan reference, and conditions.
  \item \crd{UpgradePlan}: source assessment reference, decision, risk level, score, summary, required non-mutating actions, recommended order, generation timestamp, and conditions.
\end{itemize}

Findings and evidence are embedded structured fields. They are not standalone CRDs. This is an important fidelity point because a paper that claims separate \crd{UpgradeFinding} or \crd{UpgradeEvidence} resources would not match the implementation.

\subsection{Reconciliation Flow}
The controller watches \crd{UpgradeAssessment} resources and owns generated \crd{UpgradePlan} resources. Reconciliation marks the assessment running, executes selected checkers, sorts findings deterministically, computes score and decision, upserts an idempotent plan, and then writes completed assessment status. The controller is a Kubernetes operator in the sense of the Kubernetes operator pattern: it links custom resources to controller behavior without modifying Kubernetes itself \cite{kubernetesOperatorPattern,kubernetesCustomResources,cncfOperatorWhitepaper}.

\subsection{Plan Generation}
The planner converts non-informational findings into required actions. Each action includes severity, category, resource reference, remediation text, and evidence references. The default recommended order is: fix critical blockers, fix high-risk workloads, validate admission policies, validate observability, and proceed with controlled upgrade. The order is deterministic and non-executing.

\subsection{Scope Boundaries}
The artifact scope is intentionally bounded:
\begin{itemize}[leftmargin=*]
  \item controlled benchmark metrics are produced by the external reproducibility package, not by the controller implementation itself;
  \item the removed-API checker uses an MVP static table rather than a complete versioned database;
  \item managed-provider evidence collection for AKS node-pool upgrade settings is outside the current controller;
  \item add-on support-matrix reasoning for cert-manager, ingress controllers, policy engines, storage drivers, and observability stacks is outside the core checker set;
  \item explanation-agent behavior is outside the evaluated artifact;
  \item automatic upgrade, drain, patch, or remediation execution is outside the design.
\end{itemize}

\subsection{Bounded-Write Semantics}
The phrase ``read-only'' is precise only with respect to assessed cluster resources. The operator does write its own CR status and \crd{UpgradePlan} resources. A stricter description is bounded-write assessment: no assessed workload, node, PDB, policy, webhook, CRD, add-on, or managed-provider resource is mutated by the assessment.

Controlled recall and false-positive evidence are reported separately in Section~\ref{sec:controlled-results}; production-oriented claims require the broader evaluation protocol in Section~\ref{sec:evaluation-protocol}.

\section{Controlled Benchmark Results}
\label{sec:controlled-results}
This section reports the first executed empirical slice. The benchmark is intentionally controlled: it is designed to test whether the current artifact detects labeled upgrade-readiness findings in reproducible Kind scenarios, not whether the system generalizes to all production clusters.

\subsection{Benchmark Setup}
The benchmark package is archived under \code{experiments/kind/r01-benchmark}. The final validated run is \code{results/20260617T083259Z}. It used Kind image \code{kindest/node:v1.24.15} to allow creation of Kubernetes APIs removed by later releases, target Kubernetes version \code{1.32}, Pluto \code{5.24.0}, and kubent \code{0.7.3}. The run created labeled scenarios for deprecated APIs, workload availability, readiness probes, PDB risks, admission webhook risks, and policy risks. Capacity and observability were intentionally disabled in this benchmark and remain outside the measured result.

The ground truth contains 18 expected findings: 3 deprecated-API findings, 2 workload-availability findings, 1 readiness finding, 4 PDB findings, 3 admission-webhook findings, and 5 policy-risk findings. Raw outputs include the \crd{UpgradeAssessment}, generated \crd{UpgradePlan}, baseline reports, normalized findings, metrics, controller logs, and tool-version records.

\subsection{RQ3: Baseline Coverage}
Pluto and kubent correctly matched the three deprecated-API findings in both file and cluster modes. They did not detect the 15 non-API findings because those findings concern readiness probes, replica assumptions, PDB semantics, admission webhook configuration, and policy risk. This result supports a narrow claim: API-deprecation tools are necessary baselines but do not cover the broader readiness model evaluated here. It does not prove that \tool{} is better in all upgrade workflows.

\subsection{Error Analysis}
The final controlled run produced no \tool{} false positives or false negatives against the benchmark labels. During benchmark construction, the harness exposed two implementation defects before the final measurement. First, the controller did not register \code{apiextensions.k8s.io/v1} in its runtime scheme, which caused CRD-listing checkers to fail in a live cluster. Second, the deprecated-API checker initially treated all objects served through a legacy endpoint as deprecated, even when their source manifest used the modern API. The final artifact fixes these defects by registering the CRD scheme and requiring last-applied source evidence for deprecated-API findings. A third precision issue was avoided by treating empty admission namespace selectors as broad scope, matching Kubernetes admission semantics.

\subsection{Interpretation Limits}
This benchmark is deliberately small and controlled. It does not include production workloads, managed-provider upgrade settings, resource-cost instrumentation, expert actionability ratings, or scoring calibration. The result is sufficient to answer the reviewer objection that the manuscript had no empirical result beyond unit tests. It is not sufficient to claim production effectiveness, statistically robust accuracy, or calibrated risk prediction.

\subsection{Research Questions}
\begin{description}[leftmargin=*,style=nextline]
  \item[RQ1.] Which upgrade-readiness risk families occur in controlled and representative Kubernetes workloads?
  \item[RQ2.] How accurately does \tool{} detect risk instances compared with ground truth in controlled scenarios?
  \item[RQ3.] Which risk families are missed by API-deprecation-focused baselines such as Pluto and kubent?
  \item[RQ4.] Are generated \crd{UpgradePlan} actions correct, actionable, prioritized, and supported by evidence according to expert reviewers?
  \item[RQ5.] In a future extension, does a constrained explanation layer affect review quality without introducing unsupported recommendations?

\subsection{Expanded Controlled Benchmark}
The first controlled benchmark contains 18 labeled findings and is reported in Section~\ref{sec:controlled-results}. Before a full journal submission, the benchmark should be expanded to 30 to 50 Kind scenarios across the risk families in Table~\ref{tab:risk-taxonomy}. Each scenario must include manifests, target Kubernetes version, injected risk mechanism, expected findings, and ground-truth labels. Scenarios should include single-risk and multi-risk cases. The benchmark must continue to archive \tool{} output, baseline output, and scripts so reviewers can reproduce the comparison.

\subsection{Baselines}
Pluto and kubent should be used as narrow API-deprecation baselines, not as broad readiness baselines \cite{pluto,kubent}. Helm API migration documentation and the Helm mapkubeapis plugin should be used for Helm-release API migration context \cite{helmKubernetesApis,helmMapKubeApis}. A manual platform-engineering checklist should serve as the broad operational baseline. Comparisons must be stratified by risk family; a single global F1 score against API-deprecation-only tools would be misleading.

\subsection{Managed AKS Validation}
The managed validation should use at least one AKS cluster and preferably two: a minimal cluster and a platform-style cluster with ingress, cert-manager, monitoring, a policy engine, GitOps, and the operator installed. The environment description must record AKS Kubernetes version, node image state, node pools, upgrade strategy, surge configuration, add-ons, and relevant provider settings \cite{aksUpgradeOptions,aksNodeImage,aksDay2}. This validation measures transferability, not universal generalization.

\subsection{Optional Production or Pre-Production Study}
If access is available, 3 to 5 anonymized production or pre-production clusters should be assessed in bounded-write mode. Data collection must avoid secrets, workload payloads, tenant-identifying labels, image registry credentials, and organization-specific names. Expert platform engineers should review findings and plans using a fixed rubric.

\subsection{Metrics and Expected Evidence Artifacts}
The expected evaluation artifacts are:
\begin{itemize}[leftmargin=*]
  \item per-risk-family precision, recall, F1, false positives, and false negatives;
  \item risk-family coverage compared with API-deprecation tools;
  \item assessment runtime and manual-checklist effort;
  \item expert-rated actionability, correctness, prioritization, evidence support, and remediation safety;
  \item RBAC-denied evidence gaps and their effect on decision quality;
  \item for any future explanation-layer extension, unsupported recommendation rate and evidence-grounding rate.
\end{itemize}

\section{Discussion}
\subsection{Why Evidence Matters}
Evidence changes the review conversation. A finding such as ``PDB may block disruption'' is more useful when it points to the PDB name, selector, matched workload, replica count, and exact \code{minAvailable} or \code{maxUnavailable} value. Similarly, an RBAC-denied list call is not merely a tool error; it is evidence that the assessment cannot establish readiness for a resource family. This makes uncertainty visible.

\subsection{Where the Current Artifact Is Useful}
The current artifact is useful as a repeatable pre-upgrade inspection framework, a structured way to expose evidence and assessment gaps, and a basis for comparing broader readiness risk categories against API-deprecation tools. The controlled benchmark shows that the artifact can detect the labeled risks in the current scenario package. Its value is strongest when teams want a Kubernetes-native record of findings and proposed actions without granting the tool authority to mutate workloads.

\subsection{Where the Current Artifact Is Weak}
The deprecated-API checker uses a small static MVP table. The observability checker is heuristic. The capacity checker is conservative and does not model all scheduler, autoscaler, topology, taint, priority, or managed-provider behaviors. Policy analysis is not a full admission dry run. Add-on and operator compatibility are not yet deeply analyzed. The score is not calibrated by production outcomes. These weaknesses are acceptable for an artifact-scope paper only if they remain explicit.

\subsection{Cautious Agent Integration}
\kagent{} is not part of the current evaluated artifact; any future use must be assessed through an ablation study against deterministic findings. A defensible future design would pass deterministic findings and evidence into a constrained explanation agent, require citation of evidence IDs, prohibit new finding creation, and measure unsupported recommendations. Agent output should never approve an upgrade.

\section{Threats to Validity}
\textbf{Construct validity.} Readiness is multi-dimensional. Precision and recall over labeled findings may not capture business criticality, workload ownership, maintenance-window constraints, or organizational risk tolerance.

\textbf{Internal validity.} Checker defects, incomplete API tables, incorrect benchmark labels, or baseline misconfiguration can bias results. Mitigations include independent scenario review, command archival, deterministic fixtures, and version-pinned tools.

\textbf{External validity.} The controlled Kind benchmark demonstrates behavior on labeled scenarios, not production generality. AKS validation would add managed-cluster evidence but remains provider-specific. Production clusters may include proprietary controllers, custom admission chains, and hidden operational constraints.

\textbf{Conclusion validity.} No production effectiveness claim should be made until sample size, expert-review protocol, resource-cost measurement, and statistical analysis are sufficient. The current local test suite supports implementation sanity, and the controlled benchmark supports only bounded scenario-level accuracy.

\textbf{Agent validity.} If \kagent{} is later evaluated, the study must separate deterministic evidence quality from agent explanation quality. Unsupported agent output must count against the agent layer even if the text appears plausible.

\section{Related Work}
\subsection{Software Evolution and Change Impact Analysis}
The framing of upgrades as software evolution builds on Lehman's view of long-lived software change and later software evolution work \cite{lehman1980,mens2008}. The evidence-guided model is closest to change impact analysis, where the goal is to identify what a change may affect before applying it \cite{bohner1996}. \tool{} applies this idea to a live Kubernetes cluster by treating the target Kubernetes version as the change and cluster resources as potentially affected artifacts.

\subsection{DevOps, SRE, and Production Readiness}
Continuous delivery and DevOps research provide the operational context: organizations seek faster change while preserving reliability, but adoption introduces technical and organizational challenges \cite{humbleFarley2010,chen2015,shahin2017,smeds2015,leite2019,forsgren2018}. SRE and production-readiness literature emphasizes reliability, operational feedback, explicit failure-mode handling, and review before risky operations \cite{beyer2016,nygard2018}. \tool{} follows this tradition by producing reviewable evidence rather than executing remediations.

\subsection{Infrastructure-as-Code and Kubernetes Configuration Analysis}
% TODO add/verify references for configuration drift in Kubernetes and cloud-native environments.
IaC research shows that infrastructure definitions are software artifacts with defects, security risks, and testing needs \cite{rahman2019iac}. Empirical work on Kubernetes manifests reports configuration and security misconfiguration patterns in real repositories \cite{rahman2023k8s,zhang2025k8s}. Recent drift and reconciliation work also frames infrastructure state divergence as a measurable software-engineering problem \cite{iacDriftAgents2025}. These results support the need for assessment tools that reason about concrete configuration and workload state before platform changes.

\subsection{Kubernetes API Deprecation and Upgrade Tooling}
% TODO add/verify references for empirical studies of Kubernetes upgrade failures, if available.
Kubernetes official documentation defines the authoritative semantics for API removals, version skew, eviction, PDBs, resource requests, and managed upgrade mechanics \cite{kubernetesDeprecationPolicy,kubernetesDeprecationGuide,kubernetesVersionSkew,kubernetesEviction,kubernetesDisruptions,kubernetesResourceManagement,aksUpgradeOptions,aksNodeImage,aksDay2}. Pluto and kubent specialize in deprecated API detection \cite{pluto,kubent}. Helm documentation and mapkubeapis address Helm-specific API migration issues \cite{helmKubernetesApis,helmMapKubeApis}. \tool{} differs by treating API removals as one risk family among several upgrade-readiness risks.

\subsection{Kubernetes Operators and Operator Reliability}
Kubernetes custom resources and the operator pattern define how domain-specific control loops can extend cluster behavior \cite{kubernetesCustomResources,kubernetesOperatorPattern,cncfOperatorWhitepaper}. Empirical operator-bug studies show that operators can fail through incorrect state observation, reconciliation logic, CRD definitions, and access control \cite{xu2024operatorbugs}. This motivates the manuscript's implementation-fidelity emphasis: an operator paper must state precisely which resources it reads, which resources it writes, and which controller behaviors are locally tested.

\subsection{Policy-as-Code and Admission Control}
% TODO add/verify references for policy-as-code analysis and admission-control failure studies.
Admission controllers, admission webhooks, ValidatingAdmissionPolicy, Pod Security Admission, and RBAC are central to upgrade readiness because recreated or migrated objects may be rejected even when their API versions still exist \cite{kubernetesAdmissionControllers,kubernetesAdmissionWebhooks,kubernetesValidatingAdmissionPolicy,kubernetesPodSecurityAdmission,kubernetesRbac,kubernetesRbacGoodPractices}. The Kubernetes admission-control threat model further emphasizes that admission components are security-sensitive operational dependencies \cite{kubernetesAdmissionThreatModel}. Policy-as-code systems such as OPA make policy enforcement programmable, and recent empirical work studies how policy-as-code is adopted in open-source projects \cite{opa,policyAsCodeAdoption2026}. \tool{} currently performs scoped policy and admission checks; it does not implement a full admission dry run or a complete policy-engine semantics model.

\subsection{AIOps and LLM-Based Operations Assistance as Future Context}
% TODO add/verify references for cloud-native production incidents involving upgrade or admission failures.
AIOps research studies operational automation, incident management, and failure management at scale \cite{dang2019,notaro2021}. LLM-for-software-engineering surveys and empirical security studies show both opportunities and risks: AI assistance can help with explanation and synthesis, but it may also produce unsupported or insecure output and encourage overconfidence \cite{hou2024,perry2023,pearce2022,fu2025}. This motivates the paper's cautious stance on \kagent{}: an agent layer should be optional, evidence-bounded, non-authoritative, and separately evaluated.

\section{Conclusion}
This paper reconstructs Kubernetes upgrade readiness as evidence-guided impact analysis over live cluster state. \tool{} implements this idea as a Kubernetes-native bounded-write operator that materializes assessments and plans through \crd{UpgradeAssessment} and \crd{UpgradePlan}. The current artifact includes deterministic checkers, structured evidence, scoring, decisions, and non-executing remediation actions. Local validation completed with \code{go test ./...} and 0 failures; the JSON test run reported 27 passing test events including subtests. The controlled Kind benchmark reported 18 true positives, 0 false positives, and 0 false negatives for \tool{} against the current labeled scenario package, while Pluto and kubent covered only the three deprecated-API findings. The broader protocol specifies how to extend this evidence through expanded scenarios, managed AKS validation, expert plan review, privacy controls, resource-cost measurement, scoring-sensitivity analysis, and strict ablation before any \kagent{} claim. The reviewer-facing standard is simple: every claim must be traceable either to implemented artifact behavior, measured evaluation evidence, or explicitly marked future work.
