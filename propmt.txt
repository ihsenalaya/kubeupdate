## ÉTAPE 1 — Provisionner l'infrastructure AKS réaliste (Chantier 2)

Pourquoi c'est décisif : L'accès AKS existe. C'est le seul chantier purement technique
parmi les trois décisifs. Ne pas le faire est inexcusable.

### 1.1 — IaC Terraform pour 3 clusters AKS

Créer terraform/aks-evaluation/main.tf avec trois modules :
- Cluster A (aks-small) : 1 node Standard_B2s — smoke test de régression
- Cluster B (aks-medium) : 3 nodes Standard_D2s_v3 — test multi-node readiness
- Cluster C (aks-policy) : 3 nodes Standard_D2s_v3 + Azure Policy activé + Gatekeeper managé

Chaque cluster doit avoir :
- kubernetes_version configurable (paramètre variable)
- node_count et vm_size en variables
- Output kube_config sauvegardé dans ~/.kube/config-aks-{a,b,c}
- Tags : environment="research", project="kubeupgrade-guardian"

Fichiers à créer :
- terraform/aks-evaluation/main.tf
- terraform/aks-evaluation/variables.tf
- terraform/aks-evaluation/outputs.tf
- terraform/aks-evaluation/README.md (commandes init/plan/apply/destroy + estimation de coût)

Vérification :
  terraform -chdir=terraform/aks-evaluation validate
  Attendu : "Success! The configuration is valid."

### 1.2 — Script de déploiement des workloads réels

Créer experiments/aks/scripts/deploy-real-workloads.sh qui installe :
  helm install ingress-nginx ingress-nginx/ingress-nginx --version 4.10.0
  helm install cert-manager jetstack/cert-manager --version v1.14.5 --set installCRDs=true
  helm install kube-prometheus-stack prometheus-community/kube-prometheus-stack --version 58.5.0
Plus : 3 Deployments avec PDB réels (minAvailable=1), 1 HPA, 1 StatefulSet avec PVC.

Fichiers à créer :
- experiments/aks/scripts/deploy-real-workloads.sh
- experiments/aks/scripts/teardown-workloads.sh
- experiments/aks/workloads/pdb-realistic.yaml
- experiments/aks/workloads/statefulset-with-pvc.yaml

### 1.3 — Runner d'assessment multi-cluster

Créer experiments/aks/scripts/run-assessment.sh qui :
1. Applique un UpgradeAssessment avec targetVersion paramétrable
2. Attend la phase Completed (timeout 300s)
3. Exporte kubectl get upgradeassessment -o json > findings-{cluster}-{timestamp}.json
4. Mesure : durée d'attente (secondes), RSS du pod controller (kubectl top pod)

Fichiers à créer :
- experiments/aks/scripts/run-assessment.sh
- experiments/aks/r03-aks-medium/assessment.yaml (manifest UpgradeAssessment)
- experiments/aks/r04-aks-policy/assessment.yaml

9.  deployment-two-replicas-with-pdb.yaml      (safe — ne doit PAS être flaggé)
10. statefulset-pvc-no-storageclass.yaml       (capacité problème)
11. deprecated-api-ingress-v1beta1.yaml        (API removed en 1.22)
12. deprecated-api-cronjob-v1beta1.yaml        (API removed en 1.25)
13. rbac-missing-list-permission.yaml          (RBAC gap — assessment incomplet)

18. capacity-no-headroom.yaml                  (resource requests = limits du node)
19. observability-no-probes-no-metrics.yaml    (heuristique observabilité)
20. mixed-healthy-cluster.yaml                 (cluster sain — 0 findings attendus)

### 2.2 — Instructions de labeling pour expert externe

Créer experiments/kind/r10-independent-labels/LABELING-INSTRUCTIONS.md :
- Décrire les 6 familles de risque (workload, pdb, webhook, policy, capacity, api)
- Pour chaque fixture : remplir expected-findings-{fixture-name}.yaml
- Règle : ne pas regarder le code source des checkers avant de labeler
- Fournir le niveau de confiance : "certain", "likely", "uncertain"

Créer experiments/kind/r10-independent-labels/expected-findings-template.yaml :
  fixture: "nom-du-fichier.yaml"
  labeled_by: "[NOM EXTERNE]"
  labeled_at: "[DATE ISO]"
  findings:
    - type: "PDB_NO_MATCHING_PODS"
      severity: "High"
      confidence: "certain"
  negative_control: false  # true si 0 findings attendus

### 2.3 — Script de comparaison labels vs output opérateur

Créer experiments/kind/r10-independent-labels/compare-labels.py :
- Entrée : répertoire expected-findings/*.yaml + findings JSON de l'opérateur
- Calcule TP, FP, FN, precision, recall, F1 par famille de risque
- Calcule Cohen's kappa sur les labels de sévérité (si plusieurs labeleurs)
- Output : experiments/kind/r10-independent-labels/results-summary.json

## ÉTAPE 3 — Vraie étude de scalabilité (remplace R02 incomplet)

Pourquoi c'est décisif : R02 a des p95 non monotones (7.1s→1.8s→8.7s) et s'arrête
à 1000 objets. Il faut 4 tailles, 10 runs chacune, métriques complètes.

### 3.1 — Générateur de charge déterministe

Créer experiments/scale/generate-load.sh :
  Usage : ./generate-load.sh <N> <namespace>
  Génère N Deployments (1 replica, readinessProbe présente) + N PDBs (minAvailable=1)
  Nommés load-deploy-0001 à load-deploy-N (avec zero-padding)
  Idempotent : utilise kubectl apply, pas create

### 3.2 — Runner de scalabilité avec métriques complètes

Créer experiments/scale/run-scale-study.sh :
- Paramètres : SIZES=(100 1000 5000 10000) RUNS=10
- Pour chaque taille et chaque run :
  1. kubectl create namespace scale-test-N (si absent)
  2. ./generate-load.sh N scale-test-N
  3. Enregistrer timestamp_start
  4. kubectl apply -f assessment-scale.yaml && attendre phase=Completed (timeout 600s)
  5. Enregistrer timestamp_end
  6. kubectl top pod -l app=kubeupgrade-guardian > rss.txt
  7. Lire delta requêtes API depuis /metrics du controller (si exposé)
  8. Écrire experiments/scale/r04-scale-N/run-I.json avec le format :
     {"size": N, "run": I, "duration_s": X.X, "rss_mib": Y, "api_requests_delta": Z,
      "completed": true/false, "errors": [...]}
  9. kubectl delete upgradeassessment --all -n scale-test-N

### 3.3 — Agrégateur de résultats

Créer experiments/scale/aggregate-results.py :
- Lit tous les run-*.json dans r04-scale-*/
- Calcule pour chaque taille : median, p95, p99, mean, std (durée + RSS + API requests)
- Vérifie monotonie : signale si median(100) > median(1000) etc.
- Output : experiments/scale/r04-scalability-summary.json avec :
  {"sizes": [100,1000,5000,10000], "monotonicity_ok": true/false, "anomalies": [...],
   "metrics": {"100": {"median_s":..., "p95_s":..., "rss_mib_median":...}, ...}}

Vérification :

  bash -n experiments/scale/generate-load.sh       → OK
  bash -n experiments/scale/run-scale-study.sh     → OK
  python3 -m py_compile experiments/scale/aggregate-results.py → OK

## ÉTAPE 4 — Restructurer l'article par RQ

Pourquoi c'est décisif : Un article Q1 doit pointer chaque section de résultats vers
une RQ explicite. La structure actuelle ne le fait pas.

### 5.1 — Ajouter les labels \label{rq:*} dans l'article

Dans article/kubeupgrade-guardian-readiness.tex, dans le \paragraph{Research Questions.},
ajouter \label{} après chaque \item[RQ*] :

  \item[RQ1 --- Detection Coverage.]\label{rq:detection}
  \item[RQ2 --- Comparative Value.]\label{rq:comparison}
  \item[RQ3 --- Actionability.]\label{rq:actionability}
  \item[RQ4 --- Scalability.]\label{rq:scalability}
  \item[RQ5 --- Managed Kubernetes Portability.]\label{rq:portability}

### 5.2 — Restructurer la section d'évaluation

Remplacer \section{Local Artifact Validation} et \section{Controlled Evaluation Results}
par une unique \section{Evaluation}\label{sec:evaluation} avec 5 sous-sections :

  \subsection{RQ1 --- Detection Coverage}\label{sec:rq1}
    Résultats R01 (Kind conformance). Référencer \ref{rq:detection}.
    Honnêteté : "ground truth authored by checker team — measures specification consistency"
    Pointer vers r10-independent-labels comme travail futur qui brisera la circularité.

  \subsection{RQ2 --- Comparative Value}\label{sec:rq2}
    Table par famille de risque (colonnes : outil, famille couverte oui/non).
    Baselines actuelles : Pluto, kubent (API only). Baselines à venir : Polaris, kube-score.

  \subsection{RQ3 --- Actionability}\label{sec:rq3}
    Remplacer RQ3 par une évaluation objective basée sur des métriques mesurables :
    - Nombre d'étapes dans le plan généré
    - Présence de ressource Kubernetes nommée pour chaque action
    - Présence d'une evidence record pour chaque finding
    - Taux de findings avec recommendation non vide
    Ces métriques sont calculables depuis les UpgradePlan JSON sans opinion humaine.

  \subsection{RQ4 --- Scalability}\label{sec:rq4}
    Résultats R04 quand disponibles. En attendant : données R02 avec disclaimer explicite
    sur la non-monotonie et l'absence de 5000/10000 objets.

  \subsection{RQ5 --- Managed Kubernetes Portability}\label{sec:rq5}
    Résultats AKS actuels (r02). Pointer vers r03/r04 AKS multi-cluster comme travail futur.

## REMARQUE — SÉVÉRITÉ ET RIGUEUR (ajoutée par ihsen alaya)

Cet article est destiné à un journal de niveau Q1 (IEEE TSE, EMSE, JSS, ou équivalent).
Les relecteurs Q1 rejettent sans appel les contributions qui :

1. Présentent des scripts sans données d'exécution réelles.
2. Marquent une étape "faite" parce qu'un fichier existe — sans résultats mesurables.
3. Ajoutent des expériences qui s'auto-neutralisent dans leur propre description.
4. Présentent des résultats parfaits (0-FP/0-FN) sans ground truth indépendante.
5. Gonflent le volume d'artefacts sans augmenter la preuve scientifique.

**Règle de qualification stricte — non négociable :**

- FAIT ✅ = fichier de RÉSULTATS présent dans le dépôt + données réelles + vérification passée
- EN COURS ⏳ = scripts présents mais exécution non encore faite
- À FAIRE ❌ = rien n'existe encore

Un script, un template, un scaffold = EN COURS ⏳, jamais FAIT ✅.
Les données d'exécution réelles = seule preuve acceptable.

**L'auditeur (moi) ne doit jamais être complaisant. Chaque passe doit être aussi sévère
que le relecteur le plus exigeant d'un journal Q1.**
